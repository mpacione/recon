# UI Audit — 2026-04-12

**Status:** All 8 root causes shipped on `v2` branch (commits `2638688` → `f9b7f60`, 2026-04-12 → 2026-04-13). Visually verified via snapshot renders 2026-04-13.

## Systemic Root Causes

Every individual bug traces back to one of 6 architectural patterns that need fixing at the root, not per-screen.

---

### RC-1: Chrome layout wastes 40% of every screen

**Symptoms:**
- Activity feed shows "no activity yet" on 6 of 9 screens
- Log pane shows raw INFO lines that duplicate activity feed
- Bottom chrome (activity + log + keybinds) takes ~15 lines on a 40-line terminal
- Actual content is pushed up and cramped on every screen

**Root cause:** The `ReconScreen` base class always renders all 4 chrome pieces (header, activity feed, log pane, keybind hints). There's no per-screen control over which chrome elements are relevant.

**Architectural fix:** Make chrome context-aware:
- **Activity feed**: only visible on screens where engine events fire (Run, Dashboard after a run). Hidden on Welcome, Describe, Discovery, Template, Confirm.
- **Log pane**: hidden by default everywhere. Available via a toggle keybind (`l` to show/hide logs) for debugging.
- **Keybind hints**: always visible (1 line, low cost).
- **RunStatusBar**: only on Run screen.

Implementation: `ReconScreen` subclasses already have `show_log_pane`, `show_activity_feed` flags. Set them to `False` by default, `True` only on RunScreen and DashboardScreen.

---

### RC-2: No consistent component library for selection UI

**Symptoms:**
- Template screen uses full-width 3-line centered Buttons as checkboxes
- Discovery screen uses a DataTable for toggle selection
- Confirm screen uses Buttons as radio options
- Each screen invented its own selection pattern with different sizing, alignment, and interaction

**Root cause:** No shared "selectable list item" widget. Each screen built its own from Textual primitives (Button, Static, DataTable) with inconsistent results.

**Architectural fix:** Create 2 shared widgets in `tui/widgets.py`:
1. **`ChecklistItem`** — compact 1-line toggle row. Left-aligned. Amber `[x]` or dim `[ ]`. Click to toggle. Used by: Template sections, Discovery candidates (alternative to DataTable).
2. **`RadioItem`** — compact 1-line radio option. `●` selected, `○` unselected. Click to select. Used by: Confirm model selection.

Both should be 1 line tall (not 3-line Button chrome), left-aligned, with clear color differentiation between selected (cream/amber) and unselected (dim gray).

### RC-3: No flow context or progress indicator

**Symptoms:**
- User doesn't know they're on step 3 of 6
- No breadcrumb showing Describe → Discovery → Template → Confirm → Run → Results
- No "what happens next" hint on any screen
- Screens feel disconnected — each one is an island

**Root cause:** The flow was wired screen-to-screen via callbacks in `app.py` but no shared state tracks "where am I in the flow."

**Architectural fix:** Add a flow progress component to the header bar or below it:
```
Step 3 of 6 — Research Template
describe → discovery → [template] → confirm → run → results
```
The active step is highlighted amber, completed steps are green, upcoming are dim. This sits in the `ReconScreen` chrome and updates based on which screen class is active.

Implementation: a `FlowProgress` widget in `shell.py` that maps screen class → step number. The v2 flow screens (Describe, Discovery, Template, Confirm, Run, Results) each declare their step index.

### RC-4: Static hardcoded content where dynamic content is needed

**Symptoms:**
- "ADD YOUR OWN" examples reference 3D printers (hardcoded for Bambu Lab)
- Template pre-selects only 2 sections regardless of domain
- Section pool key mismatch (`pricing_business` vs `pricing`) causes selection bugs
- `rounds: 0` when search is in progress (should say "round 1 in progress")
- `1 rounds complete` (grammar)
- `API ✓` shows green for invalid keys (checks existence, not validity)

**Root cause:** Content is hardcoded at build time instead of derived from runtime state.

**Architectural fix:**
- Template examples: generate from the domain description via a simple string template or remove entirely (the placeholder text is sufficient).
- Section pre-selection: fix the key mismatch, then use the `schema_designer.design_sections()` LLM call we already built to make intelligent selections.
- Round counter: show state accurately ("searching..." during, "N complete" after).
- API key validation: add a lightweight ping test on save (HEAD request to Anthropic API).

### RC-5: Layout/sizing issues from Textual widget defaults

**Symptoms:**
- Worker `+`/`-` buttons broken horizontal layout (+ button off-screen)
- Competitor names truncated ("Amazon Q Developer (form...")
- Describe screen is still a ModalScreen (centered box) not a full screen
- Logo box border clips on welcome screen
- Scrollbar was Textual default blue (fixed) but there may be other default-style leakage
- Center-aligned button text looks odd for checklist items

**Root cause:** Textual's default widget sizing, alignment, and layout assumptions don't match our design intent. Each case needs explicit CSS overrides.

**Architectural fix:**
- **Standard width classes**: define `.compact-btn` (1-line, left-aligned, transparent), `.action-btn` (standard button), `.primary-btn` (amber highlight) in `theme.py` so every screen uses the same sizing.
- **Name truncation strategy**: define a max-width constant for competitor names. Truncate from the parenthetical first ("Amazon Q Developer" not "Amazon Q Developer (form..."). Add a `truncate_name(name, max_width)` utility.
- **Promote Describe to ReconScreen**: it's the only flow screen still a ModalScreen.
- **Fix worker +/- layout**: use explicit width constraints on the Horizontal container.

### RC-6: Redundant information with no clear hierarchy

**Symptoms:**
- Header shows `$0.00 · 0 runs` during an active run (misleading)
- Run screen shows `0/28` in header AND `0/2` per competitor (redundant)
- Worker numbers `[W1]`, `[W2]` are meaningless to users
- No estimated time remaining anywhere
- No per-call cost visible in the main content area (only in activity feed)
- `idle` state shown in header when nothing is happening (wasted space)

**Root cause:** Information was added incrementally without an information architecture pass. Each widget shows what it knows without considering what other widgets already show.

**Architectural fix:**
- **Header bar**: show only: `recon | Company · Domain | path | $cost | state`. Drop `runs` count (not useful mid-flow). Drop `idle` label (show nothing when idle). During a run, show `running · stage` with live cost.
- **Run screen**: the global progress bar (`0/28`) is the primary metric. Per-competitor fractions are secondary detail. Remove worker numbers — just show `CompetitorName · section  elapsed  ⠹`.
- **Add ETA**: `estimated_remaining = (elapsed / completed) * remaining` displayed in the header: `~4 min remaining`.
- **Cost per competitor**: show cumulative cost in the left column next to each competitor name.

## Fix Sequence

### RC-7: Content scrolls away from controls

**Symptoms:**
- Discovery: buttons at top scroll away when viewing candidates below
- Template: Proceed/Back buttons scroll off-screen when scrolled to bottom sections
- Confirm: action buttons may scroll off on smaller terminals
- General: user has to hunt for the "next" action after reviewing content

**Root cause:** Action buttons and content are in the same scrollable container. No fixed/sticky button bar.

**Architectural fix:** Pin action buttons to a fixed footer within each screen's body area. Content above scrolls freely; buttons stay visible at all times. Implementation: wrap buttons in a `Vertical` with `dock: bottom` inside the body area, or use Textual's `Footer`-style docking within the compose_body layout.

---

### RC-8: Insufficient contextual detail at decision points

**Symptoms:**
- Confirm doesn't name which sections were selected (just "2 sections")
- Confirm doesn't show estimated wall-clock time
- Confirm model descriptions don't explain capability differences
- Discovery header doesn't show candidate count
- Template has no "Select All" / "Deselect All"
- Recent projects show no completion status (researched? abandoned?)
- No label on the `n` new project path input
- Path display uses absolute `/Users/...` not `~/`

**Root cause:** Each screen shows minimal data to make its own decision. Context from previous screens is lost. The user is expected to remember what they selected 2 screens ago.

**Architectural fix:** Each decision screen should be self-contained:
- Confirm: list the selected section names, show "~4 min estimated", explain model tiers in 1 line each
- Discovery: show "14 found" in the header, show count on the Done button ("Done (14 accepted)")
- Template: add Select All / Deselect All buttons, show count of selected in header
- Welcome: show last-run date and status per recent project, use `~/` paths
- New project input: add a "Save to:" label above the input field
