# TUI Audit — 2026-04-10 (post Phase E/F/G/H/I)

**Method.** Drove the recon TUI through every screen via the
`/tmp/recon-audit/capture.py` script (updated for the keybind flow),
captured 25 SVG + plain-text snapshots, read each one, noted what
worked and what didn't. Then re-captured after the critical fixes
landed.

**Inputs.** The capture script:
- Starts on the welcome screen with no recents
- Re-runs with two recent projects
- Mounts the new-project Input via `action_new()` and the
  open-project Input via `action_open()`
- Walks the wizard step-by-step through identity → sections →
  sources → API key
- Renders the dashboard empty and populated (5 competitors)
- Pushes the planner, picks UPDATE_SPECIFIC, walks into the selector
- Pushes the browser, then Discovery (with no API key, then with
  3 candidates injected)
- Triggers the manual-add flow on Discovery
- Pushes the curation gate with 3 themes
- Switches to the run mode (idle, then with bus events fired for
  RunStarted, 3 SectionResearched, 3 CostRecorded, RunCompleted)

**Output.**
- 25 SVG screenshots in `/tmp/recon-audit/svg/`
- 25 plain-text renders in `/tmp/recon-audit/txt/` (terminal-like
  text reconstructed from SVG `<text>` elements)
- 4 critical fixes shipped in commit `f04ff28` (Phase I)

## Findings, by severity

### SEV-1 (broken UX, fixed in Phase I)

| # | Finding | Status |
|---|---|---|
| 1 | Planner buttons rendered with no labels (since Phase A). `height: 1` clipped Textual Button content rows; users saw 7 empty boxes and a hint "press 1-7" with no indication of what each number did. Snapshot baselines had been "passing" with the broken render. | **FIXED** — `height: 3` on `.operation-row` |
| 2 | Selector `[x]` checkbox markers silently dropped. Rich markup parsed `[x]` as an unknown tag. Selected items showed only the name with no marker. | **FIXED** — escape open bracket as `\[x][/]` |
| 3 | Theme curation `[x]` markers had the same Rich markup bug. | **FIXED** — same escape |
| 4 | Chrome header bar didn't update from engine bus events (Phase C feature was silently broken). `_on_engine_event` only used `call_from_thread`, which raises `RuntimeError` on the message-loop thread. The exception was caught in a bare `except` and silently dropped. Audit caught this when the header still showed `$0.00 / 0 runs / idle` after publishing 3 `CostRecorded` events. | **FIXED** — `_schedule_chrome_refresh` tries inline first, falls back to `call_from_thread` only on worker threads |

### SEV-2 (cosmetic / clarity, deferred)

| # | Finding | Notes |
|---|---|---|
| 5 | Welcome banner ASCII art truncates at the right edge of the 70-char container in some terminals. The ▄/█/▀ characters wrap to a second line. | Cosmetic. Could fix by widening the welcome container or shrinking the banner. |
| 6 | Wizard modals have no chrome — no header, no Esc hint, no log pane. By design (Phase B opt-out for modals), but users have no documented way to back out of the wizard without completing it. | Add an "Esc to cancel" hint to each wizard step. |
| 7 | Discovery says "Click Search More to find competitors" — mouse-first language for a keyboard-driven TUI. | One-line copy fix. |
| 8 | Selector and curation modals scale poorly when there are many items because each Button takes 3 lines (border + content + border). 5 competitors = 15 lines + container padding. With 20 items, the modal exceeds the screen. | Move to Static-row rendering with custom click/key handlers, or wrap the list in a scrollable container. |
| 9 | The TUI body has two sources of truth for the active run: the reactive attrs on `RunScreen` (set by `pipeline_runner.py`) and the `workspace_context` mutated by the bus subscriber. Both are now correct as of Phase I, but they rely on the pipeline runner doing both updates in lockstep. | Plumb the body widgets to read from `workspace_context` so the bus is the single source of truth. |
| 10 | Recents file leaks across script/test runs because `~/.recon/recent.json` is the canonical store. Tests that don't isolate this dir bleed entries into the welcome screen. | Already mitigated in tests via fixture; capture script could swap to a tmp recent path. |
| 11 | Path display in the chrome header uses `humanize_path` which collapses tmp dirs to `$TMP/...` but the result is still 25-30 chars. With long workspace paths, the ws label dominates the header strip. | Consider truncating to 20 chars with an ellipsis when the header is space-constrained. |
| 12 | The empty welcome state and the dashboard empty prompt both use static "press X" instructional text. There's no visual indication that the keybind hint strip at the bottom of the chrome is the canonical place to learn what each screen does. New users will look for buttons. | One-time onboarding hint, or a faint arrow pointing to the keybind strip. |

### Layout / structural

| # | Finding | Notes |
|---|---|---|
| 13 | Chrome takes ~17 fixed lines of vertical space (1 header + 1 status bar + 8 ActivityFeed + 8 LogPane + 1 KeybindHint, minus 1 status bar when idle). On a 24-line terminal, the body has only ~6 lines. | Adaptive sizing — collapse ActivityFeed/LogPane on small terminals, or make them scrollable in a split. |
| 14 | The `Vertical#recon-footer` wrapping fix (Phase H) was the right call but it means the body region's `height: 1fr` competes with the footer's `height: auto`. On very tall terminals the body has tons of empty space, on short terminals the footer takes everything. | Could give the footer a bounded height (e.g., `max-height: 18`) so the body has a guaranteed minimum. |
| 15 | The selector's title `── SELECT COMPETITORS ──` doesn't show in the plain-text render, but IS in the SVG. This is a snapshot-rendering artifact (the SVG-to-text script drops bold styling), not a real bug. | Audit script limitation. |

### Brittleness

| # | Finding | Notes |
|---|---|---|
| 16 | TUI unit tests use the `yield Screen` pattern instead of `push_screen`. This causes the screen to mount as a child widget, which means `Screen.BINDINGS` only fire if a focusable child has focus (Textual's binding traversal walks ancestors of the focused widget). With buttons gone, unit tests bypass via `screen.action_xxx()` direct calls, which works but doesn't actually test the binding declaration. End-to-end binding behavior only verified by `test_tui_integration.py` and `test_tui_real_terminal.py`. | The integration + smoke tests catch real binding issues, so this is OK in practice. |
| 17 | `_render` is a reserved method name on every Textual Widget. Static subclasses that define `_render` silently break the visual pipeline (Visual is None, AttributeError on render). I tripped over this twice this session (Phase B for LogPane, Phase G for RunStatusBar). | Add a project-level lint check that warns on `def _render(` inside Static subclasses. |
| 18 | `[x]` (and any other 1-char identifier in brackets) renders as nothing in Rich markup. Bare `[/]` is valid (closes a tag) but `[x]` is parsed as an unknown tag. | Add a project-level lint check or a Rich-markup-safe label helper. |
| 19 | Snapshot tests can "pass" with broken rendering forever, because the baseline IS the broken render. The dashboard/run/welcome snapshots were ~14KB blank canvases for two phases before Phase H caught it. The planner snapshot was missing button labels for many phases before Phase I caught it. | Adopt a snapshot pre-commit check: snapshots should contain a minimum number of expected text markers (e.g., dashboard snapshot must contain "COMPETITORS" and "discover"). Or generate the snapshots fresh on every commit and require manual review of diffs. |
| 20 | Welcome's `RecentProjectsManager` writes to `~/.recon/recent.json` directly, not under the workspace dir. This bleeds across users / projects / test runs. | Add per-workspace scoping or honor an env var override. |

## Reflection on the flow

### What works well

- **Persistent chrome.** The header bar + ActivityFeed + LogPane + RunStatusBar + keybind hint stack gives the user constant context. Phase B/C/F/G/I work paid off — the user always knows where they are, what's running, what just happened, and what they can press.
- **Keybind-first navigation.** Phase E removed the action-bar buttons in favor of single-letter keys. After the planner labels were fixed in Phase I, the keyboard flow is genuinely fast: press `r` → press `1`-`7` → run.
- **Real engine wiring.** From Phase A's audit forward, the TUI actually calls the pipeline. The bus + reactive header + ActivityFeed + RunStatusBar + RunScreen body all reflect engine state in real time.
- **TDD discipline.** Every change in this session went test-first. The 691-test suite caught a lot, and the smoke test caught what the unit tests missed.

### What feels brittle

- **Two sources of truth for run state.** The `RunScreen` reactive attrs and the `workspace_context` are kept in sync by both the pipeline runner *and* the bus subscriber writing to them. They agree today but the coupling is fragile.
- **Modal vs full-screen split.** Modals (wizard, planner, selector, curation, discovery) opt out of chrome. Users lose the header, the activity feed, and the keybind hint when they're inside a modal. They have to remember they're 1 Esc away from the main screen.
- **Rich markup is a footgun.** The `[x]` bug bit twice (selector + curation) and is invisible until you actually look at the rendered output. Same for `_render` overrides. The static-typed parts of the codebase are great; the dynamic UI strings are landmines.

### What can confuse users

- **Wizard has no exit.** No keybind, no documented Esc, no Cancel button on intermediate steps. Users have to power through 4 steps or kill the app.
- **Empty dashboard "press d to discover" but no API key.** If the user has no `ANTHROPIC_API_KEY` set, pressing `d` shows the discovery screen with an inert "Search More" button. The hint that they need an API key only appears as a notification toast that disappears.
- **Cost preview vs running cost vs total cost.** The planner shows "est. full run: $0.21". The chrome header shows "$1.09" (running). The dashboard shows "$X.XX across N runs" (total). Three different costs in three different places, all called "cost".
- **Discovery's "Click Search More" copy** is mouse-first language in a keyboard-first TUI.

### What's general enough

- **Schema-driven sections.** Adding a new section to `recon.yaml` requires no code changes; the wizard, dashboard, planner, and pipeline all read from the schema. ✓
- **Multi-mode workspace context.** Each workspace gets its own state.db, recon.yaml, .env. The TUI handles workspace switching cleanly. ✓
- **Pipeline operations.** The 7 planner ops cover the realistic CI workflows: full run, update all, update specific, diff all, diff specific, rerun failed, add new. Discovery is a separate flow. ✓

### What's NOT general enough

- **Anthropic-only LLM.** Hardcoded to `claude-sonnet-4-5` everywhere. No abstraction for other providers. If the user wants to try Gemini or GPT-5 they'd need to fork the engine.
- **English-only prompts.** All prompts in `prompts.py` are English. Non-English domains would need a translation layer.
- **Single-machine local.** ChromaDB + SQLite are local. No multi-user, no cloud sync, no team workspace.
- **Dollars only.** Cost rendering is hardcoded to `$X.XX`. EUR/GBP/JPY users see dollar signs.

These are scope decisions, not bugs — recon is intentionally a single-user local CLI. Worth noting because the architectural plan needs to decide whether to invest in cross-cutting abstractions or stay focused.
